## 2. Hotfix dispatcher — spread defensivo (v3.0.5)

**Motivação:** log do user em prod mostrou `ws.tipo_sem_handler tipo=mais_vendidos`.
Sintoma idêntico ao hotfix v3.0.3, mas causa diferente: tarefas LEGADAS
em status PENDENTE no DB (criadas antes do v3.0.3 com `"tipo"` no payload)
ainda eram re-entregues quando o agente reconectava, e o spread sobrescrevia.

**Fix em `app/services/dispatcher.py:_tentar_entrega`:**

```python
# Antes:
payload = {"tipo": comando, "tarefa_id": tarefa.id, **tarefa.payload}
# ↑ payload PODE sobrescrever tipo/tarefa_id se tiver essas chaves

# Depois:
payload = {**(tarefa.payload or {}), "tipo": comando, "tarefa_id": tarefa.id}
# ↑ tipo/tarefa_id SEMPRE ganham
```

Loga `tarefa.payload_chave_conflitante` quando detecta caso legado.

Mesmo fix aplicado em `app/services/busca_service.py:_entregar_para_agente`.

**Commit:** [a0b7d61](https://github.com/silviosvargas/achadinhosv3/commit/a0b7d61)

**Motivação:** o user pediu pra fazer "igual à V2" — extrair produtos E
gerar meli.la no MESMO driver Chrome, antes de salvar. A V3 separou em

**Refatoração:**

`agente/agent/busca_ml.py`:
- Nova função `_gerar_meli_la_no_driver(driver, produtos, log_prefixo)`
  que recebe driver aberto + lista de produtos, abre painel linkbuilder
  ML NO MESMO driver, captura meli.la, atualiza
  `produto["url_afiliado"]` in-place.
- Chamada de `_varrer_lista_urls_sync`, `_varrer_termo_livre_sync` e
  `_varrer_produto_unico_sync` ANTES do `finally` que fecha o driver.
- Cf V2 `src/buscar/ml.py:executar()` chamava `gerar_todos_links(driver, novos)`
  no mesmo driver da busca.

`app/services/busca_service.py:_upsert_produto`:
- Detecta se `item.url_afiliado` veio como `meli.la/XXX` (agente novo) e
  salva direto. Fallback genérico `?matt_word=...` só se o agente não
  conseguiu gerar.

  fez tudo inline; servidor não precisa orquestrar segunda tarefa.

**Vantagens vs fluxo anterior:**
1. Sem conflito de driver (1 só Chrome ML rodando, sempre)
2. Sem race condition de re-entrega WS após reconexão
3. Sem tarefa GERAR_LINK órfã em PROCESSANDO

## 10. BUG RAIZ — handler retornava sem `ok=True` (v3.0.10)

**Motivação:** mesmo com tudo acima, `produtos.url_afiliado` continuava
com fallback. Investigação minuciosa do `ws_client.py` mostrou:

```python
# agente/agent/ws_client.py:144-157
if resultado.get("ok"):
    await self.enviar({"tipo": "tarefa_concluida", ...})
else:
    await self.enviar({"tipo": "tarefa_falhou", "erro": ...})
```

E `agente/agent/main.py:handler_gerar_links_ml`:
```python
return {"mapping": mapping, "total": len(mapping)}  # ← FALTA "ok": True
```

**Resultado:** desde a Fase 15 (v3.5.0) até a v3.0.9, GERAR_LINK SEMPRE
caía no `else` → servidor recebia `tarefa_falhou` → marcava tarefa como
FALHOU → `dispatcher.marcar_concluida` **nunca** chamava
`afiliado_ml_writer.aplicar_mapping` → `produtos.url_afiliado`
**nunca** virava `meli.la`.

Por isso o ML gerava os shortlinks corretamente (logs:
`gerados=10 pedidos=10`) mas o DB ficava intacto. **Os 6 fixes anteriores
(v3.0.4 a v3.0.9) eram melhorias reais mas nenhum atacava esse bug porque
o callback nem chegava no servidor.**

**Fix em `agente/agent/main.py`:**
```python
return {
    "ok":      True,        # ← essencial pro ws_client
    "mapping": mapping,
    "total":   len(mapping),
    "pedidos": len(urls),
}
```

Documentado o contrato dos handlers em
[contrato_handlers_ws.md](contrato_handlers_ws.md) pra próximos handlers
não caírem na mesma armadilha.

**Commit:** [1a76992](https://github.com/silviosvargas/achadinhosv3/commit/1a76992)

## Lições registradas

1. **Handler WS sem `ok=True` é falha silenciosa.** Sempre adicionar
   contrato no template. Ver [contrato_handlers_ws.md](contrato_handlers_ws.md).
2. **Quando algo "rodou mas não salvou", investigar o callback WS antes
   de mexer na lógica de domínio.** A sequência v3.0.4 → v3.0.9 atacou
   o problema errado por 6 releases.
3. **Comparar V2 vs V3 quando V2 funcionava.** A V2 era mais simples
   (1 driver, 1 fluxo). Replicar a simplicidade quando possível.
4. **URL canônica do ML SEMPRE deve ser limpa na extração** (V2 style).
   Não confiar em camadas posteriores pra normalizar.
5. **`reentregar_pendentes` não deve incluir PROCESSANDO.** Tarefa em
   execução real, se WS cair no meio, o callback chega quando WS subir
   de novo. Re-entregar = execução duplicada → conflito de Chrome.
6. **Cache de 5min em endpoints que mudam por release é demais.** Reduzir
   pra 60s ou expor `?nocache=1`.
7. **URL canônica do ML tem variantes:** `/p/MLB1234567890` (catálogo
   simples), `/up/MLBU3387021403` (catálogo unificado). Regex precisa
   aceitar `MLB[A-Z]?-?\d{8,15}` se for match flexível.
